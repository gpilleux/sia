## REQUIREMENT DESCRIPTION (REVISED)

Implement **robust multi-tenant SaaS architecture** with:
1. **Company-level data isolation** - Enable multiple clients without data collision
2. **Demo sandbox** - Public interactive demo (company with `is_demo=TRUE`)
3. **Authentication** - Admin login to access real company data
4. **Scalability** - Easy onboarding of new clients (2-minute setup)

**Critical Context**: 
- Current state: Purchase module IS multi-tenant (company_id), Sales/CRM are NOT
- Inconsistency: 71 endpoints public, NO authentication
- Goal: Align all modules to multi-tenant pattern + add auth layer

### Problem to Solve
1. **Architectural Inconsistency**: Purchase module is multi-tenant, Sales/CRM are NOT
2. **Data Collision Risk**: Cannot onboard new clients (no company isolation)
3. **Demo Requirements**: Need public sandbox WITHOUT affecting real client data
4. **Security Gap**: Zero authentication - all data publicly accessible
5. **Scalability Blocker**: Cannot scale to SaaS model without multi-tenancy

### Expected Value
- **Business**: Enable rapid client onboarding (2-minute setup per new company)
- **Security**: Database-level data isolation (FK constraints enforce company_id)
- **Demo**: Interactive sandbox resets nightly, zero impact on clients
- **Scalability**: Foundation for unlimited companies (SaaS-ready)
- **Robustness**: Industry-standard multi-tenant pattern (Odoo, Salesforce, ERPNext)

## INVARIANTS TO SATISFY

Mathematical/logical constraints that MUST be true:

- [ ] **INV-1 (Multi-Tenancy)**: `∀ entity ∈ {Customer, Product, Order, Lead, Opportunity, Invoice} → entity.company_id ∈ Companies.id (FK CONSTRAINT)`
- [ ] **INV-2 (Data Isolation)**: `∀ query(user) → results = {e | e.company_id = user.company_id}` (auto-filtered)
- [ ] **INV-3 (Demo Sandbox)**: `∃! company WHERE company.is_demo = TRUE ∧ company.id = '00000000-0000-0000-0000-000000000000'`
- [ ] **INV-4 (Authentication)**: `∀ endpoint ∈ /api/v1/* → (authenticated(request) → company_id = JWT.company_id) ∨ (¬authenticated(request) → company_id = DEMO_COMPANY_ID)`
- [ ] **INV-5 (JWT)**: `JWT → {sub: user_id, company_id: UUID, exp: timestamp} ∧ exp ≤ now() + 24h`
- [ ] **INV-6 (Cross-Tenant Block)**: `user.company_id = A ∧ entity.company_id = B → access_denied(user, entity)` (404 NOT 403)

### Architectural Constraints

**DDD Principles**:
- Company is natural Aggregate Root (multi-tenancy boundary)
- Each entity belongs to exactly one Company (FK constraint)
- Purchase module already follows this pattern (reference implementation)
- Cross-module consistency: ALL modules use same multi-tenant pattern

**SOLID Principles**:
- **SRP**: Repository filters by company_id (single responsibility)
- **OCP**: Adding new company does NOT modify existing code
- **LSP**: All repositories implement `find_by_company(company_id)`
- **ISP**: Company-scoped interfaces (no god repositories)
- **DIP**: Domain depends on `IRepository`, infrastructure implements filtering

**KISS Principles**:
- ✅ Reuse existing multi-tenancy (Purchase module pattern)
- ✅ Standard SaaS architecture (Odoo, Salesforce, ERPNext)
- ✅ Database-level enforcement (FK constraints)
- ❌ NO custom permission system (company_id is the permission)
- ❌ NO complex ACLs (tenant isolation is binary: you see your company or you don't)

## ADDITIONAL NOTES

### Security Considerations
- **Critical**: Current ERP is FULLY OPEN (71 endpoints, no auth). This is CRITICAL security risk.
- **JWT Secret**: Must use environment variable (NOT hardcoded)
- **Password Storage**: NEVER store plaintext (use bcrypt with salt)
- **CORS**: Already configured for localhost:5173 and erp.appstart.cl
- **SQL Injection**: Use SQLAlchemy ORM (already doing this) - NO raw SQL

### Performance Considerations
- **Rate Limiting**: 5 req/min per IP for public endpoint (prevent spam)
- **JWT Validation**: Add caching for user permissions (avoid DB query per request)
- **Password Hashing**: bcrypt is slow by design (~200ms) - acceptable for login

### Edge Cases
- **Expired JWT**: Return 401, frontend redirects to login
- **User Deleted**: JWT remains valid until expiry (acceptable for MVP)
- **Role Changed**: User must logout/login to refresh permissions
- **Concurrent Requests**: JWT is stateless (no race conditions)

### Migration Strategy
- **Backward Compatibility**: NO BREAKING CHANGES to domain/entities
- **Data Migration**: Add default role "EMPLOYEE" to existing users in init.sql seed data
- **Frontend Migration**: Gradual replacement of DEBUG_USERS (keep for dev mode)
