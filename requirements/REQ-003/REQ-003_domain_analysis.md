## EXECUTIVE SUMMARY

### MCP-UI Protocol Overview

**MCP-UI** es un SDK que estandariza cómo las aplicaciones MCP (Model Context Protocol) pueden generar y renderizar UI interactiva. Permite que el backend (Python) genere componentes visuales declarativamente y el frontend (React) los renderice automáticamente en iframes sandboxed.

**Fit con Argus**: 
- ✅ **Problem-Solution Fit**: Resuelve exactamente nuestro problema de renderizar visualizaciones dinámicas
- ✅ **Architecture Alignment**: Separa generación (backend) de renderizado (frontend) → DDD compliant
- ✅ **Security**: Sandboxing nativo evita XSS → crítico para plataforma gubernamental
- ✅ **Scalability**: Protocol-based, permite agregar nuevos tipos de visualizaciones sin breaking changes

### ⚠️ CRITICAL DISCOVERY: Google ADK Gap Analysis

**Investigation Findings** (2025-11-24):

Durante la investigación arquitectónica se identificó una **discrepancia crítica** entre la documentación (`.sia/agents/argus.md`) y el código en producción:

**Documentación SPR** (`.sia/agents/argus.md`):
```markdown
## CHAT SSE PIPELINE (REQ-002, QUANT-027, ...)

7. ADK RUNNER EXECUTION (REQ-002):
   ↓
   Initialize PostgresAdkSessionService (Adapter)
   ↓
   Initialize Agent (gpt-4o-mini) with Tools (Viz + Data)
   ↓
   Runner.run_async(user_id, session_id, message)
```

**Código Real** (`backend/src/api/v1/chat.py:394-398`):
```python
# Step 3: Execute multi-agent workflow
from infrastructure.adk.orchestrator import \
    execute_multi_agent_workflow

result = await execute_multi_agent_workflow(user_query=request.message, max_retries=2)
```

**Gap Identified**:
| Component        | Documented State                        | Actual State                          | Impact                      |
| ---------------- | --------------------------------------- | ------------------------------------- | --------------------------- |
| Chat Endpoint    | Uses `Runner.run_async()`               | Uses `execute_multi_agent_workflow()` | ❌ No ADK Runner             |
| Orchestrator     | ADK `LlmAgent`                          | Custom async function                 | ❌ No tool calling framework |
| Tools            | `BaseTool` implementations              | Plain async functions                 | ❌ No artifact generation    |
| Session Service  | `PostgresAdkSessionService` active      | Only in demos/tests                   | ❌ No ADK persistence        |
| Artifact Service | Available via `context.save_artifact()` | Not accessible                        | ❌ **BLOCKER for MCP-UI**    |

**Root Cause**: REQ-002 was documented as "ADK Integration Complete" but actual implementation is **boilerplate/POC only**, not production-ready.

**Implication for REQ-003**: MCP-UI `UIResource` objects require `google.genai.types.Part` artifacts from ADK Artifact Service. Custom orchestrator cannot generate these artifacts.

**Architectural Decision**: Implement **PHASE 0** to migrate chat endpoint to Google ADK Runner **BEFORE** integrating MCP-UI.

### Justification for ADK Migration

**Why ADK Runner is Essential**:

1. **Artifact Service** (`context.save_artifact()`):
   - Manages binary content (HTML, images, PDFs) as `types.Part` objects
   - Provides lifecycle management (create, retrieve, cleanup)
   - MCP-UI adapter converts `types.Part` → `UIResource`

2. **Tool Calling Framework**:
   - Standardized `BaseTool` interface with `input_schema` + `output_schema`
   - Automatic validation and error handling
   - LLM can dynamically choose tools based on context

3. **Session Management**:
   - `PostgresAdkSessionService` handles persistence automatically
   - History reconstruction for multi-turn conversations
   - No manual message saving required

4. **Event Streaming**:
   - ADK Runner yields structured events (`text`, `tool_call`, `artifact`)
   - Clean mapping to SSE events
   - Built-in error handling and retries

**Dependency Chain**:
```
MCP-UI UIResource
    ↓ (requires)
google.genai.types.Part (ADK Artifact)
    ↓ (generated by)
context.save_artifact()
    ↓ (available in)
BaseTool.run_async(context, ...)
    ↓ (executed by)
Runner.run_async()
```

**Conclusion**: Cannot implement MCP-UI without ADK Runner. Migration is **mandatory prerequisite**.

## DOMAIN MODEL: GOOGLE ADK

### Core Entities

#### 1. UIResource (Value Object)
```typescript
interface UIResource {
  uri: string;           // Unique ID (must start with "ui://")
  mimeType: string;      // Determines rendering strategy
  text?: string;         // HTML/JavaScript/URL content
  blob?: string;         // Base64 encoded binary content
  metadata?: {           // Optional client hints
    "mcpui.dev/ui-preferred-frame-size"?: { width: number, height: number },
    "mcpui.dev/ui-initial-render-data"?: any
  }
}
```

**Invariants**:
- `uri` MUST start with `ui://`
- `mimeType` ∈ {`text/html`, `text/uri-list`, `application/vnd.mcp-ui.remote-dom+javascript`}
- Exactly ONE of `text` or `blob` must be present (XOR)

#### 2. UIAction (Command Object)
```typescript
interface UIAction {
  type: 'tool' | 'intent' | 'prompt' | 'notify' | 'link';
  payload: {
    toolName?: string;
    params?: Record<string, any>;
    intentName?: string;
    message?: string;
    url?: string;
  }
}
```

**Invariants**:
- `type === 'tool'` → `payload.toolName` MUST exist
- `type === 'link'` → `payload.url` MUST exist and be valid URL
- Actions flow FROM iframe TO host application

### Content Types (3 Strategies)

#### Strategy 1: Raw HTML (`text/html`)
**Use Case**: Self-contained visualizations (charts, tables, cards)

```python
# Backend (Python)
create_ui_resource({
    "uri": "ui://argus/chart/bar",
    "content": {
        "type": "rawHtml",
        "htmlString": """
            <div style="padding: 20px;">
                <h2>Document Distribution</h2>
                <div id="chart"></div>
                <script src="https://cdn.jsdelivr.net/npm/recharts@2.x"></script>
                <script>
                    // Chart rendering logic
                </script>
            </div>
        """
    },
    "encoding": "text"
})
```

**Rendering**: Iframe with `srcDoc` attribute (no network request)

**Pros**:
- ✅ Offline-first (no external dependencies)
- ✅ Full control over HTML/CSS/JS
- ✅ Fast rendering (no HTTP roundtrip)

**Cons**:
- ❌ Large HTML strings increase payload size
- ❌ Inline JS can be verbose

#### Strategy 2: External URL (`text/uri-list`)
**Use Case**: Embed external services (Grafana, Tableau, custom apps)

```python
create_ui_resource({
    "uri": "ui://argus/external/grafana",
    "content": {
        "type": "externalUrl",
        "url": "https://grafana.example.com/dashboard/123?embed=true"
    },
    "encoding": "text"
})
```

**Rendering**: Iframe with `src` attribute

**Pros**:
- ✅ Leverage existing dashboards
- ✅ No HTML generation needed

**Cons**:
- ❌ Requires network access
- ❌ External service availability dependency
- ❌ CORS/CSP considerations

#### Strategy 3: Remote DOM (`application/vnd.mcp-ui.remote-dom+javascript`)
**Use Case**: Host-native components (use frontend's component library)

```python
create_ui_resource({
    "uri": "ui://argus/component/native-table",
    "content": {
        "type": "remoteDom",
        "scriptContent": """
            import { h, render } from '@remote-dom/core';
            const Table = h('table', {}, ...rows);
            render(Table, document.body);
        """
    },
    "encoding": "text"
})
```

**Rendering**: Executes in iframe, sends JSON diffs to host, host renders with native components

**Pros**:
- ✅ Native look & feel (uses host's Tailwind/Shadcn)
- ✅ Performance (host renders, not iframe)

**Cons**:
- ❌ More complex (requires @remote-dom library)
- ❌ Learning curve steeper
- ❌ Not needed for simple HTML visualizations

from mcp_ui_server import create_ui_resource

class VisualizationFactory:
    """Factory for creating UIResource objects from domain data."""
    
    @staticmethod
    def create_bar_chart(data: list[dict], title: str) -> UIResource:
        html = f"""
        <!DOCTYPE html>
        <html>
        <head>
            <script src="https://cdn.tailwindcss.com"></script>

                const data = {json.dumps(data)};
                // Recharts rendering logic
            </script>
        </body>
        </html>
        """
        
        return create_ui_resource({
            "uri": f"ui://argus/chart/bar/{uuid.uuid4()}",
            "content": {"type": "rawHtml", "htmlString": html},
            "encoding": "text",
            "uiMetadata": {
                "mcpui.dev/ui-preferred-frame-size": {"width": 800, "height": 400}
            }
        })

### Pattern 2: Client-Side Rendering (React SDK)

```tsx
import { UIResourceRenderer, UIAction, UIActionResult } from '@mcp-ui/client';

function ChatMessage({ message }: { message: Message }) {
  const handleUIAction = async (action: UIAction): Promise<UIActionResult> => {
    switch (action.type) {
      case 'tool':
        // Execute tool on backend
        const result = await fetch('/api/v1/tools/execute', {
          method: 'POST',
          body: JSON.stringify({
            toolName: action.payload.toolName,
            params: action.payload.params
          })
        });
        return { status: 'handled', data: await result.json() };
      
      case 'prompt':
        // Insert message into chat
        sendMessage(action.payload.message);
        return { status: 'handled' };
      
      case 'link':
        // Navigate to URL
        window.open(action.payload.url, '_blank');
        return { status: 'handled' };
      
      default:
        return { status: 'unhandled' };
    }
  };

  return (
    <div className="message">
      <p>{message.content}</p>
      {message.uiResources?.map((resource) => (
        <UIResourceRenderer
          key={resource.uri}
          resource={resource}
          onUIAction={handleUIAction}
          htmlProps={{
            autoResizeIframe: true,
            style: { border: '1px solid #334155', borderRadius: '8px' }
          }}
        />
      ))}
    </div>
  );
}
```

## SECURITY MODEL

### Iframe Sandboxing

**MCP-UI Default Sandbox Attributes**:
```html
<iframe 
  sandbox="allow-scripts allow-same-origin"
  srcdoc="..."
  style="border: none; width: 100%; height: 400px;"
></iframe>
```

**What's Allowed**:
- ✅ JavaScript execution (for charts/interactivity)
- ✅ Same-origin requests (within iframe context)

**What's Blocked**:
- ❌ Form submission (`allow-forms` not set)
- ❌ Top-level navigation (`allow-top-navigation` not set)
- ❌ Popups (`allow-popups` not set)
- ❌ Pointer lock (`allow-pointer-lock` not set)

### Communication Protocol

**Iframe → Host (postMessage)**:
```javascript
// Inside iframe (user clicks button)
window.parent.postMessage({
  type: 'tool',
  payload: {
    toolName: 'get_more_data',
    params: { filter: 'recent' }
  }
}, '*');
```

**Host → Iframe (postMessage)**:
```javascript
// Host sends data to iframe
iframe.contentWindow.postMessage({
  type: 'update_data',
  payload: { newData: [...] }
}, '*');
```

**MCP-UI Mediator**: Automatically validates and transforms messages

### CSP Requirements

```http
Content-Security-Policy: 
  default-src 'self'; 
  script-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net https://cdn.tailwindcss.com; 
  frame-src 'self' blob:; 
  style-src 'self' 'unsafe-inline';
```

**Rationale**:
- `script-src` includes CDN for recharts/tailwind
- `unsafe-inline` needed for inline scripts in `srcDoc`
- `blob:` needed if using blob URLs for iframes

### SOLID Principles Application

**Single Responsibility**:
- `BarChartGenerator`: Solo genera HTML de bar charts
- `TableGenerator`: Solo genera HTML de tablas
- `UIResourceFactory`: Solo crea UIResource objects

**Open/Closed**:
- Factory pattern permite agregar `LineChartGenerator`, `HeatmapGenerator` sin modificar código existente
- Nueva visualización = nueva clase que implementa `BaseGenerator`

**Liskov Substitution**:
- Cualquier `BaseGenerator` puede ser usado por `VisualizationGenerator` sin conocer implementación concreta

**Interface Segregation**:
- `IChartGenerator`: `generate(data: ChartData) -> str`
- `ITableGenerator`: `generate(data: TableData) -> str`
- Clientes solo dependen de interfaz mínima

**Dependency Inversion**:
- Application layer depende de `IVisualizationGenerator` (abstraction)
- Infrastructure inyecta `MCP_UIVisualizationGenerator` (concretion)

## PERFORMANCE CONSIDERATIONS

### Iframe Overhead

**Benchmarks** (from MCP-UI research):
- **Single Iframe**: ~5-10ms initial render
- **10 Iframes**: ~50-100ms total (parallel loading)
- **Memory**: ~2-5MB per iframe (depends on content)

**Mitigation Strategies**:
1. **Lazy Loading**: Use `IntersectionObserver` to defer off-screen iframes
2. **Virtual Scrolling**: Only render visible charts in long dashboards
3. **Resource Pooling**: Reuse iframes for similar content types

### HTML Payload Size

**Typical Sizes**:
- Bar Chart (recharts): ~8-15KB (HTML + inline JS)
- Table (TailwindCSS): ~3-5KB
- Metric Card: ~1-2KB

**Optimization**:
- Minify HTML templates
- Use CDN for libraries (cached by browser)
- Gzip compression (FastAPI default)

### Network Considerations

**SSE Payload Example**:
```json
{
  "event": "ui_resource",
  "data": {
    "uri": "ui://argus/chart/bar/abc123",
    "mimeType": "text/html",
    "text": "<html>...</html>",  // ~10KB
    "metadata": { "mcpui.dev/ui-preferred-frame-size": {"width": 800, "height": 400} }
  }
}
```

**Size**: ~10-20KB per visualization

**Mitigation**:
- Use `blob` + Base64 for binary content (smaller)
- Implement compression for large HTML
- Consider `externalUrl` for very large visualizations

## ALTERNATIVE APPROACHES EVALUATED

### Alternative 1: Direct React Components (No MCP-UI)
**Approach**: Backend sends JSON data, frontend renders with recharts directly

**Pros**:
- ✅ Type-safe (TypeScript end-to-end)
- ✅ No iframe overhead
- ✅ Native React performance

**Cons**:
- ❌ Tight coupling (backend must know frontend capabilities)
- ❌ Breaking changes when adding new chart types
- ❌ No security isolation
- ❌ Complex state management for dynamic charts

**Verdict**: ❌ **Rejected** - Violates separation of concerns, no standardization

### Alternative 2: Server-Side Rendering (HTML Strings)
**Approach**: Backend generates complete HTML, frontend uses `dangerouslySetInnerHTML`

**Pros**:
- ✅ Simple implementation
- ✅ Full backend control

**Cons**:
- ❌ XSS vulnerability (no sandboxing)
- ❌ No interactivity (JS execution blocked)
- ❌ SEO issues (dynamic content not indexed)

**Verdict**: ❌ **Rejected** - Security risk unacceptable for government platform

### Alternative 3: Web Components (Custom Elements)
**Approach**: Backend sends web component definitions, frontend registers and renders

**Pros**:
- ✅ Native browser standard
- ✅ Encapsulation (Shadow DOM)
- ✅ Reusable across frameworks

**Cons**:
- ❌ No sandboxing (XSS risk)
- ❌ Complex registration logic
- ❌ Browser compatibility issues (older browsers)
- ❌ Harder to debug

**Verdict**: ❌ **Rejected** - Similar to Alt 2 but more complex

### Selected Approach: MCP-UI
**Rationale**:
- ✅ **Security**: Iframe sandboxing prevents XSS
- ✅ **Standardization**: Protocol-based, extensible
- ✅ **Separation of Concerns**: Backend generates, frontend renders
- ✅ **Interactivity**: Full JS execution in safe context
- ✅ **Community**: Growing ecosystem (Anthropic-backed)

## TECHNICAL RISKS & MITIGATION

### Risk 1: Iframe Performance Degradation
**Probability**: MEDIUM  
**Impact**: HIGH (UX degradation)

**Mitigation**:
- Implement lazy loading (only render visible iframes)
- Use virtual scrolling for long dashboards
- Set performance budget: Max 10 iframes per view
- Monitor FCP/LCP metrics with Lighthouse

### Risk 2: CDN Availability (recharts, tailwind)
**Probability**: LOW  
**Impact**: HIGH (charts fail to render)

**Mitigation**:
- Self-host critical libraries in `/public/vendor/`
- Implement fallback logic: Try CDN → fallback to local
- Cache CDN responses with Service Worker
- Monitor CDN uptime (UptimeRobot)

### Risk 3: Browser Compatibility
**Probability**: LOW  
**Impact**: MEDIUM (some users see broken UI)

**Mitigation**:
- Test on browsers: Chrome 90+, Firefox 88+, Safari 14+
- Use polyfills for older browsers (core-js)
- Feature detection: Check `window.postMessage` support
- Graceful degradation: Show static image if iframe fails

### Risk 4: Content Size Bloat
**Probability**: MEDIUM  
**Impact**: MEDIUM (slow SSE streaming)

**Mitigation**:
- Compress HTML templates (minify)
- Use Brotli compression (FastAPI middleware)
- Implement pagination for large tables
- Set max size limit: 50KB per UIResource

### Risk 5: Learning Curve (Team Adoption)
**Probability**: HIGH  
**Impact**: LOW (delays, not blockers)

**Mitigation**:
- Create internal documentation with examples
- Pair programming sessions for first implementation
- Code review checklist for UIResource generation
- Reference implementation in `examples/` folder
