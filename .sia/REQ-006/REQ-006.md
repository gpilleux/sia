## EXECUTIVE SUMMARY

Ampliar el MCP server `repo_indexer_mcp.py` con herramientas adicionales que permitan **gestionar el ciclo de vida del índice** directamente desde VS Code Copilot, eliminando la necesidad de ejecutar comandos CLI manualmente.

**Problema Actual**: 
- MCP es READ-ONLY (solo `search_code`)
- Indexación requiere CLI manual: `cd repo_indexer && uv run repo-indexer index --repo /path --repo-name name`
- Flujo de trabajo fragmentado: CLI para indexar → MCP para buscar
- Sin visibilidad de estado (¿qué repos están indexados? ¿cuándo fue la última actualización?)

**Solución Propuesta**:
Agregar 4 herramientas MCP:
1. `index_repository` - Indexar/re-indexar repositorio
2. `get_indexed_repos` - Listar repos con stats (chunk_count, last_indexed)
3. `delete_index` - Eliminar índice de un repo
4. `get_indexing_status` - Estado de indexación en progreso (async task tracking)

**Resultado**: Super Agent puede gestionar completamente el knowledge graph sin intervención manual del usuario.

**New Tools Required**:

#### 1. `index_repository` (WRITE operation)
```python
@mcp.tool
async def index_repository(
    repo_path: str,
    repo_name: str,
    force_reindex: bool = False
) -> dict:
    """
    Index or re-index a repository.
    
    Args:
        repo_path: Absolute path to repository (e.g., /Users/.../sia)
        repo_name: Identifier for search (e.g., "sia")
        force_reindex: If True, delete existing index first
    
    Returns:
        {
            "task_id": "uuid",
            "status": "started",
            "repo_name": "sia",
            "estimated_files": 120
        }
    """
    # Delegate to IndexRepositoryUseCase
    use_case = IndexRepositoryUseCase(
        vector_store, 
        file_metadata_repo, 
        ast_analyzer, 
        embedding_service
    )
    
    # Run async background task
    task_id = await use_case.execute_async(repo_path, repo_name, force_reindex)
    return {"task_id": task_id, "status": "started"}
```

#### 2. `get_indexed_repos` (READ operation)
```python
@mcp.tool
async def get_indexed_repos() -> list[dict]:
    """
    List all indexed repositories with statistics.
    
    Returns:
        [
            {
                "repo_name": "sia",
                "chunk_count": 587,
                "last_indexed": "2025-11-25T10:30:00Z",
                "file_count": 120,
                "avg_chunk_size": 450
            }
        ]
    """
    # Query vector_store + file_metadata_repository
    repos = await vector_store.list_repositories()
    return repos
```

#### 3. `delete_index` (DELETE operation)
```python
@mcp.tool
async def delete_index(repo_name: str) -> dict:
    """
    Delete all indexed data for a repository.
    
    Args:
        repo_name: Repository identifier
    
    Returns:
        {
            "repo_name": "sia",
            "chunks_deleted": 587,
            "files_deleted": 120
        }
    """
    async with AsyncSessionLocal() as session:
        chunks_deleted = await session.execute(
            delete(CodeChunkModel).where(CodeChunkModel.repo_name == repo_name)
        )
        files_deleted = await session.execute(
            delete(FileMetadataModel).where(FileMetadataModel.repo_name == repo_name)
        )
        await session.commit()
    
    return {
        "repo_name": repo_name,
        "chunks_deleted": chunks_deleted.rowcount,
        "files_deleted": files_deleted.rowcount
    }
```

#### 4. `get_indexing_status` (READ operation - async task tracking)
```python
@mcp.tool
async def get_indexing_status(task_id: str) -> dict:
    """
    Check status of async indexing task.
    
    Args:
        task_id: Task identifier from index_repository
    
    Returns:
        {
            "task_id": "abc-123",
            "status": "processing",  # or "completed", "failed"
            "progress": {
                "files_indexed": 45,
                "total_files": 120,
                "chunks_created": 234,
                "estimated_time_remaining": 30  # seconds
            },
            "errors": []  # List of file processing errors
        }
    """
    # Query in-memory task tracker or Redis/database
    task_status = await task_tracker.get_status(task_id)
    return task_status
```

### Database Schema Additions

**New Table: `indexing_tasks`**
```sql
CREATE TABLE indexing_tasks (
    task_id UUID PRIMARY KEY,
    repo_name VARCHAR(255) NOT NULL,
```

### Async Task Management

**Background Processing** (avoid blocking MCP stdio transport):

```python
# infrastructure/services/task_manager.py
class AsyncTaskManager:
    def __init__(self):
        self._tasks: dict[str, IndexingTask] = {}
    
    async def create_task(
        self, 
        repo_name: str, 
        repo_path: Path
    ) -> str:
        task_id = str(uuid4())
        task = IndexingTask(
            task_id=uuid.UUID(task_id),
            repo_name=repo_name,
            repo_path=repo_path,
            status="started"
        )
        self._tasks[task_id] = task
        
        # Run indexing in background (asyncio.create_task)
        asyncio.create_task(self._run_indexing(task_id, repo_name, repo_path))
        
        return task_id
    
    async def _run_indexing(
        self, 
        task_id: str, 
        repo_name: str, 
        repo_path: Path
    ):
        try:
            # Update status
            self._tasks[task_id].status = "processing"
            
            # Execute IndexRepositoryUseCase
            use_case = IndexRepositoryUseCase(...)
            stats = await use_case.execute(repo_path, repo_name)
            
            # Mark completed
            self._tasks[task_id].status = "completed"
            self._tasks[task_id].chunks_created = stats["chunks_indexed"]
            self._tasks[task_id].completed_at = datetime.utcnow()
        except Exception as e:
            self._tasks[task_id].status = "failed"
            self._tasks[task_id].error_message = str(e)
    
    async def get_status(self, task_id: str) -> dict:
        task = self._tasks.get(task_id)
        if not task:
            raise ToolError(f"Task {task_id} not found")
        
        return {
            "task_id": task_id,
            "status": task.status,
            "progress": {
                "files_indexed": task.files_processed,
                "chunks_created": task.chunks_created
            }
        }
```

## RISKS & MITIGATIONS

### Risk 1: Blocking MCP Transport
**Scenario**: Large repo indexing (10k files) blocks stdio for minutes  
**Impact**: VS Code Copilot appears frozen  
**Mitigation**: Use `asyncio.create_task` for background processing, return task_id immediately

### Risk 2: Memory Overflow
**Scenario**: Indexing large repo loads all embeddings in memory  
**Impact**: MCP server crashes (OOM)  
**Mitigation**: Batch processing (100 files at a time), stream embeddings to DB

### Risk 3: Database Connection Pool Exhaustion
**Scenario**: Multiple concurrent indexing tasks consume all connections  
**Impact**: New requests fail  
**Mitigation**: Limit concurrent tasks (semaphore), increase pool size in config

### Risk 4: Task State Loss on Restart
**Scenario**: MCP server restarts, in-memory task tracker loses state  
**Impact**: User can't check status of long-running indexing  
**Mitigation**: Persist tasks to `indexing_tasks` table, restore on startup

## APPENDIX: Alternative Approaches Considered

### Alternative 1: Keep CLI + Add MCP Proxy
**Approach**: MCP tool calls CLI via subprocess  
**Pros**: Reuses existing CLI logic  
**Cons**: Synchronous (blocks), fragile (depends on PATH), no progress tracking  
**Decision**: ❌ Rejected - violates async MCP architecture

### Alternative 2: Webhook-Based Indexing
**Approach**: MCP triggers webhook, FastAPI server runs indexing  
**Pros**: Clean separation, scalable  
**Cons**: Requires additional HTTP server, complex setup  
**Decision**: ⚠️ Future consideration for distributed systems

### Alternative 3: Immediate Indexing (No Async)
**Approach**: MCP tool blocks until indexing completes  
**Pros**: Simpler implementation  
**Cons**: Terrible UX (10+ min blocks), violates MCP best practices  
**Decision**: ❌ Rejected - unacceptable user experience
