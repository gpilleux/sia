## REQUIREMENT DESCRIPTION

Simplificar el proceso de creación de slash commands eliminando pasos pesados de research/QUANT cuando el output es simple y directo. El workflow actual (req.prompt.md) es excelente para requirements complejos pero overkill para crear un prompt.md de 50-100 líneas.

### Problem to Solve

Actualmente crear un slash command requiere:
- Domain analysis profundo (MCP DeepWiki obligatorio)
- Extracción de invariantes con notación matemática
- QUANT breakdown completo
- Múltiples documentos (REQ.md, domain_analysis.md, quant_breakdown.md)

Para un slash command simple (ej: `/clean` que organiza archivos), este workflow es excesivo. Necesitamos un atajo directo.

### Expected Value

- **Velocidad**: Crear slash command en 10-15 min vs 60-90 min actuales
- **Pragmatismo**: Aplicar research pesado solo cuando justificado
- **Mantener calidad**: Usar SLASH_COMMAND_DESIGN_PATTERNS.md como guía suficiente
- **Self-evolution**: Framework mejora su propia tooling

## IMPLEMENTATION APPROACH

**Nuevo Slash Command**: `/slash`

**Purpose**: Crear slash commands siguiendo SLASH_COMMAND_DESIGN_PATTERNS.md sin research pesado

**Protocol**:
1. User input → Descripción del comando deseado
2. Validar complejidad → Simple (directo) vs Complex (usa /req)
3. Aplicar patterns de SLASH_COMMAND_DESIGN_PATTERNS.md
4. Generar prompt.md (50-100 líneas)
5. Copiar a `sia/templates/prompts/` y `.sia/prompts/`
6. Actualizar SLASH_COMMANDS.md

**Distinction Logic**:
- **Simple** → UI/workspace operations, file organization, basic automation
- **Complex** → Requiere MCP research, domain modeling, architectural decisions

## CASE STUDY: `/clean` Command

**Input**: "Limpiar repositorio, archivos en ubicaciones correctas"

**Classification**: SIMPLE (workspace hygiene, no domain research)

**Output Expected** (50-70 líneas):
```markdown
---
name: clean
description: Organizar archivos del workspace en ubicaciones canónicas
argument-hint: "[--dry-run|--force]"
---

🧹 **REPOSITORY CLEANUP PROTOCOL**

Organiza workspace → Archivos en ubicaciones correctas → Metadata sincronizada.

**PROTOCOLO:**
1. Scan workspace → Detectar archivos fuera de ubicaciones canónicas
2. Clasificar → Prompts, skills, requirements, docs, configs
3. Proponer movimientos → Mostrar plan, esperar confirmación
4. Ejecutar (si aprobado) → Mover archivos, actualizar metadata
5. Validar → Verificar integridad post-cleanup

**DETECCIÓN AUTOMÁTICA:**
- `*.prompt.md` fuera de `.sia/prompts/` → Mover
- Scripts en root → Clasificar (skill vs tool)
- Docs sueltos → Consolidar en `docs/` o `.sia/knowledge/`
- Archivos temporales → Limpiar (backups, .DS_Store, __pycache__)

**PRINCIPIOS:**
- **Safety First**: Dry-run por defecto, requiere confirmación
- **Traceability**: Log de movimientos en `.sia/metadata/cleanup.log`
- **Rollback**: Backup automático antes de mover archivos

**ANTI-PATTERNS:**
- ❌ Mover archivos sin confirmación
- ❌ Modificar archivos críticos (.git, pyproject.toml)
- ❌ Borrar sin backup

**OUTPUT:**
```
✅ CLEANUP COMPLETED
📁 Archivos movidos: 5
🧹 Temporales eliminados: 12
📋 Log: .sia/metadata/cleanup.log
```
```
