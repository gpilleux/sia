## REQUIREMENT DESCRIPTION

Refactor all framework skills (`check_complexity.sh`, `visualize_architecture.sh`, `check_coverage.sh`, `audit_ddd.py`, `metrics.py`) to achieve production-grade quality, enabling reliable verification gates for QUANT lifecycle and framework dogfooding.

**Audit Evidence**: `requirements/SKILLS_AUDIT_2025-11-25.md` identified 100% failure rate in inception mode due to hardcoded assumptions, missing error handling, and false positive success messages.

### Problem to Solve

**Critical Failures**:
1. **Inception Blindness**: Skills assume DDD application structure (domain/, tests/), fail when executed on SIA framework itself
2. **Hardcoded Paths**: `uv run sia/skills/metrics.py` breaks in inception mode (skills/ at root, not sia/skills/)
3. **False Positives**: Scripts print "✅ Success" even when commands fail (no exit code checking)
4. **No Graceful Degradation**: Missing directories (tests/) cause crashes instead of informative warnings
5. **Broken Tools**: `visualize_architecture.sh` uses wrong flag (`--output` vs `-o`), never generates graphs

**Impact**: 
- Cannot use skills as QUANT verification gates (unreliable)
- Framework integrity violation (dogfooding failure)
- Misleading output wastes developer time (negative productivity)

### Expected Value

**Post-Refactoring State**:
- ✅ Skills execute successfully in both inception mode (SIA repo) and submodule mode (inherited projects)
- ✅ Accurate exit codes (0 = success, non-zero = failure)
- ✅ Clear, actionable error messages (no false positives)
- ✅ Graceful degradation (missing tools/directories = warnings, not crashes)
- ✅ Framework self-application (SIA can audit itself)
- ✅ Reliable QUANT verification gates (complexity, coverage, architecture validation)

### Architectural Constraints

**DDD Principles**:
- Skills bounded context = Pure utilities (no domain coupling)
- Skills → Core (read-only access to SUPER_AGENT.md for inception detection)
- Skills ¬→ Agents/Requirements (no circular dependencies)

**SOLID Principles**:
- **SRP**: Each skill = single responsibility (complexity XOR coverage XOR architecture)
- **OCP**: Extensible via --help flags, configuration files (future)
- **LSP**: Bash/Python scripts interchangeable where applicable
- **ISP**: Minimal interface (stdin → skill → stdout + exit code)
- **DIP**: Skills depend on abstractions (tool availability checks, not direct calls)

**Clean Architecture**:
- Skills = Outer layer (Infrastructure/Tools)
- No imports from Core/Agents (one-way dependency)
- Exception: Read-only file checks (e.g., exists(core/SUPER_AGENT.md))

**KISS Principle**:
- Prefer simple path resolution over complex configuration
- Use exit codes over JSON/YAML output formats
- Shell scripts for simple tasks, Python for complex logic

## RESEARCH NEEDED

### Domain Questions

**Deepwiki Research Completed** (2025-11-25):

1. ✅ **ShellCheck (koalaman/shellcheck)**: 
   - Exit code patterns: Direct command checks (`if command; then`) > `if [ $? -eq 0 ]`
   - Error propagation: Avoid masked exit codes (SC2312)
   - Graceful degradation: Try preferred method, fallback to alternatives
   - Cross-platform: Avoid tilde in $PATH, use portable redirection

2. ✅ **uv (astral-sh/uv)**:
   - Ephemeral deps: `uv run --with <package>` for temporary installations
   - Inline metadata: PEP 723 script metadata for self-contained scripts
   - Missing packages: Use try/except, provide installation guidance
   - Detection: Search script directories, handle UvNotFound errors

3. ✅ **pytest (pytest-dev/pytest)**:
   - Missing dirs: Exit code 5 (no tests collected), no crash
   - Collection: `pytest --collect-only` previews without execution
   - Graceful handling: Check exit codes, use --ignore for exclusions
   - Configuration: `testpaths` in pyproject.toml controls search scope

4. ✅ **CPython AST (python/cpython)**:
   - Safe parsing: Wrap `ast.parse()` in try/except for SyntaxError
   - Imports: Analyze `ast.Import` and `ast.ImportFrom` nodes
   - Decorators: Inspect `decorator_list` on ClassDef/FunctionDef
   - Best practices: Use ast.NodeVisitor for traversal, ast.unparse() for code gen

### Implementation Patterns (Evidence-Based)

**Inception Detection Pattern**:
```bash
# Bash
if [ -f "core/SUPER_AGENT.md" ]; then
    INCEPTION_MODE=true
    METRICS_PATH="skills/metrics.yaml"
else
    INCEPTION_MODE=false
    METRICS_PATH=".sia/metrics.yaml"
fi
```

```python
# Python
def is_inception_mode(root: Path) -> bool:
    return (root / "core" / "SUPER_AGENT.md").exists()
```

**Path Resolution Pattern**:
```bash
resolve_skill_path() {
    local skill=$1
    for path in "skills/$skill" "sia/skills/$skill"; do
        if [ -f "$path" ]; then
            echo "$path"
            return 0
        fi
    done
    echo ""
    return 1
}
```

**Error Handling Pattern** (ShellCheck-compliant):
```bash
if uv run --with pydeps pydeps "$TARGET" -o "$OUTPUT"; then
    echo "✅ Graph generated at $OUTPUT"
else
    echo "❌ Failed to generate graph"
    echo "   Hint: Ensure uv is installed: pip install uv"
    exit 1
fi
```

**Graceful Degradation Pattern** (pytest-inspired):
```bash
if [ ! -d "$TEST_DIR" ]; then
    echo "⚠️  Test directory '$TEST_DIR' not found"
    echo "   Hint: Create tests/ or specify: $0 . custom/test/path"
    exit 0  # Warning, not error
fi
```

**Safe AST Parsing Pattern** (CPython best practice):
```python
try:
    tree = ast.parse(source_code, filename=filepath)
except SyntaxError as e:
    print(f"⚠️  Syntax error in {filepath}: {e}")
    return  # Skip file, continue analysis
```
