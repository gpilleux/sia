## PROBLEM STATEMENT

### Current Gap
Proyectos que usan SIA como submódulo **no reciben automáticamente** las mejoras cuando se hace `git pull` del submódulo:
- Nuevos prompts en `templates/prompts/` no se sincronizan a `.sia/prompts/`
- Actualizaciones de skills en `skills/` no se propagan a `.sia/skills/`
- Mejoras de agentes no se reflejan en `.sia/agents/`

### Evolution Flow (Problemático)
```
Proyecto A (.sia/prompts/custom.md) 
  → evolve_spr.py → templates/prompts/custom.md
  → git push (submódulo SIA)
  → Proyecto B: git submodule update --remote
  → ❌ .sia/ en Proyecto B NO se actualiza
```

### Root Cause
El instalador (`install.py`) **solo se ejecuta una vez**. Los cambios posteriores en `templates/` del submódulo quedan huérfanos.

---

## TECHNICAL ARCHITECTURE

### **Synchronization Protocol (6 Fases)**

```
FASE 1: DETECCIÓN DE CONTEXTO
├── Leer sia/VERSION
├── Leer .sia/metadata/sia_version.txt
└── Decisión: bootstrap | update | downgrade | no-op

FASE 2: INVENTARIO
├── list_dir("sia/templates/prompts/")
├── list_dir(".sia/prompts/")
└── Calcular: new_prompts, existing_prompts

FASE 3: SINCRONIZACIÓN INTELIGENTE
├── NEW → create_file()
├── EXISTING → Comparar contenido:
│   ├── Idéntico → Skip
│   ├── Hash coincide → replace_string_in_file()
│   └── Hash difiere → Preguntar usuario
└── Crear backup si sobrescribe

FASE 4: OTROS COMPONENTES
├── Skills (solo nuevos)
├── Framework Agents (actualizar existentes)
└── Templates de referencia

FASE 5: ACTUALIZAR METADATA
├── .sia/metadata/sia_version.txt
├── .sia/metadata/last_sync.txt
├── .sia/metadata/original_hashes.json
└── .sia/metadata/sync.log

FASE 6: REPORTE FINAL
└── Resumen detallado para usuario
```

### **Conflict Resolution Matrix**

| Template State | Local State | Hash Match | Action |
|----------------|-------------|------------|--------|
| New | N/A | N/A | ✅ Copy (create_file) |
| Modified | Unchanged | ✅ Match | 🔄 Update (replace) |
| Modified | Modified | ❌ Mismatch | ⚠️ Ask user |
| Deleted | Exists | N/A | ℹ️ Mark orphan (keep) |

## KEY FEATURES

### **Protecciones Automáticas**
- 🔒 **SPR del proyecto** - NUNCA tocado
- 🔒 **Configuración local** (`.sia.detected.yaml`) - NUNCA sincronizada
- 🔒 **Archivos personalizados** - Detectados vía hash, preguntar antes de sobrescribir
- 🔒 **Backups automáticos** - Creados en `.sia/backup/{timestamp}/`

### **Detección Inteligente**
```python
# Hash-based personalization detection
original_hash = metadata["original_hashes"]["prompts/quant.prompt.md"]
current_hash = shasum(.sia/prompts/quant.prompt.md)

if current_hash == original_hash:
  → NO personalizado → Seguro actualizar
else:
  → PERSONALIZADO → Preguntar usuario
```

### **Variantes del Comando**

| Comando | Comportamiento |
|---------|----------------|
| `/sync` | Sincronización inteligente (interactiva) |
| `/sync --dry-run` | Mostrar cambios SIN aplicar |
| `/sync --force` | Sobrescribir TODO (requiere confirmación) |
| `/sync rollback` | Restaurar desde último backup |

## EDGE CASES HANDLED

**EC-1: Primera Sincronización**
```
If NOT exists(.sia/metadata/sia_version.txt):
  → Bootstrap mode
  → Copiar TODO sin preguntar
  → Crear metadata inicial
```

**EC-2: Downgrade Detectado**
```
If local_version > framework_version:
  → Advertir: "Submódulo más antiguo"
  → Preguntar: "¿Continuar con downgrade?"
```

**EC-3: Metadata Corrupta**
```
If parse error original_hashes.json:
  → Regenerar metadata
  → Asumir todos como potencialmente personalizados
```

**EC-4: Archivos Huérfanos**
```
If prompt en .sia/ pero NO en framework:
  → Reportar como huérfano
  → Opciones: mantener | archivar | eliminar
```

## LESSONS LEARNED

### **Tool-Based > Script-Based**
**Observation**: Inicialmente se propuso `smart_update.py` (script Python), pero se pivotó a approach tool-based.

**Razón**: 
- Super Agent YA tiene capacidad de manipular archivos vía tools
- Prompt documenta protocolo → Más mantenible que código
- UX conversacional > CLI batch
- Menos código = menos bugs

**Principio Evolucionado**: 
> "Si Super Agent puede hacerlo con tools, NO crear script. Los prompts SON el código ejecutable."

### **Hash-Based Personalization Detection**
**Approach**: Guardar hash original de cada archivo en metadata → Comparar con hash actual.

**Ventaja**: Detección precisa sin necesidad de git diff o markers manuales.

**Trade-off**: Requiere metadata inicial (bootstrap en primera sync).
